\section{Cloud Services Selection and Justification}
LightsPlease's architecture is constructed around AWS services to provide secure public access, managed device connectivity, automation, telemetry ingestion, camera streaming, operational visibility, and durable storage. The services selected here were chosen to keep the design technically attainable for the scope of the project while still aligning with the main functional and operational needs of LightsPlease.

\textbf{Amazon CloudFront} provides edge caching and content delivery for static frontend assets and other cacheable responses \cite{aws-cloudfront}. In this architecture, it improves delivery performance for the web client while fitting cleanly with the AWS-based ingress path. It was selected over a self-managed CDN because the project does not benefit enough from the added control to justify operating separate edge infrastructure. The main trade-off is less low-level configurability, but that is reasonable for a platform of this scope.

\textbf{AWS Web Application Firewall (WAF)} filters common web exploits, abusive traffic, and unwanted request patterns before they reach backend services \cite{aws-waf}. This adds a managed security layer at the public edge that complements application-level validation and reduces the need to operate a separate firewall tier. The trade-offs are the possibility of false positives and less flexibility than a fully self-managed solution, but those are acceptable given the goal of keeping the platform operationally simple.

\textbf{Amazon Cognito} provides customer identity management, token issuance, sign-in flows, and multi-factor authentication support \cite{aws-cognito}. It was selected so the platform could rely on a managed identity service rather than implementing account security, token handling, and recovery flows from scratch. That makes the authentication layer easier to integrate with the rest of the ingress path and reduces the amount of security-sensitive code that must be maintained directly. The trade-off is tighter dependence on AWS-native identity patterns, but that is preferable here to building and operating a custom authentication stack.

\textbf{Amazon API Gateway} provides the managed HTTPS entry point for client requests and supports routing, throttling, authorization hooks, and policy enforcement \cite{aws-api-gateway}. The public control plane for LightsPlease is API-driven, so API Gateway fits naturally as the boundary between the client and the backend services. It was chosen over a permanently running web tier on EC2 or ECS because that would add more infrastructure management without providing much benefit for this workload. Its main limitations are per-request pricing and service limits around payload size, timeout, and request shape.

\textbf{AWS Lambda} provides event-driven compute for stateless request handling, command validation, automation triggers, and service integration logic \cite{aws-lambda}. This is a good fit because household activity is bursty and much of the platform workload consists of short, independent tasks rather than long-running sessions. Lambda also keeps the control plane easier to scale without maintaining always-on application servers. The trade-offs are execution time limits, cold starts, and weaker support for long-lived connections, but those are acceptable for the platform's control and orchestration workloads.

\textbf{AWS Step Functions} provides managed workflow orchestration for multi-step automations that require branching, waiting, retries, and state tracking \cite{aws-step-functions}. It was chosen because customer-defined routines become harder to reason about when all workflow logic is embedded directly inside individual Lambda functions. Step Functions makes that behavior more explicit and easier to monitor, while also providing built-in retry handling for longer-running sequences. The cost is added workflow complexity and per-state-transition pricing, but that is still preferable to building a custom orchestration layer for more complex automations.

\textbf{AWS IoT Core} provides secure device messaging, device identity, and message routing for lights, locks, thermostats, sensors, and other non-video modules \cite{aws-iot-core}. It was selected because this platform depends on persistent device connectivity, secure command delivery, and cloud-managed policy enforcement for a large number of constrained devices. Using IoT Core avoids the security and maintenance burden of running a self-managed broker layer while also keeping the design aligned with MQTT-based device communication. The trade-off is tighter AWS coupling and per-message cost, but that is a reasonable exchange for managed device connectivity in this project.

\textbf{Amazon Kinesis Video Streams} provides managed ingest, buffering, and retrieval for camera feeds and other video-heavy traffic \cite{aws-kinesis-video-streams}. Cameras were given a separate media path because video has very different bandwidth, retention, and playback requirements than ordinary device telemetry. Kinesis Video Streams was chosen instead of a custom streaming backend because live camera access is a core requirement and the project does not benefit from self-managing a dedicated media stack. The trade-offs are higher cost and a more specialized integration path, but both are justified by the camera use case.

\textbf{Amazon DynamoDB} provides a managed NoSQL storage for operational state such as household records, device state, automation definitions, and command metadata \cite{aws-dynamodb}. It was selected because these reads and writes need to remain fast as the number of households and connected devices grows. DynamoDB also fits the architecture's preference for managed services that scale without a large administrative footprint. The main trade-off is that access patterns must be designed more deliberately up front and ad hoc querying is more limited than in a relational database.

\textbf{Amazon Timestream} provides managed time-series storage for telemetry such as temperature history, motion events, and power usage readings \cite{aws-timestream}. This was chosen because historical sensor data has different query and retention behavior than current operational state. Keeping time-series telemetry separate from the main state store allows the architecture to retain and query device history more efficiently without forcing that workload into a general-purpose database. The trade-off is additional architectural complexity, but it keeps the data layer better aligned to the workload.

\textbf{Amazon S3} provides durable object storage for archived logs, retained video clips, camera snapshots, and other backend files that do not fit well into the primary database layer \cite{aws-s3}. It was selected because these objects are better treated as durable files than as low-latency application state. S3 also fits naturally with the LightsPlease's archival and media retention needs without requiring dedicated file infrastructure. The trade-offs are higher retrieval latency than database-backed storage and the need to manage lifecycle, retention, and access policies carefully.

\textbf{Amazon CloudWatch} provides centralized logs, metrics, alarms, and dashboards for monitoring the health of the APIs, Lambda functions, workflows, and device integrations \cite{aws-cloudwatch}. It was selected because the LightsPlease spans multiple managed services and needs a common operational view for failures, performance issues, and alerting. CloudWatch keeps that observability path inside the same AWS environment as the rest of the architecture, which simplifies operations for a small team. The trade-off is additional dependence on AWS-native monitoring tooling, but that is consistent with the rest of the design.

\textbf{Amazon SNS} provides lightweight pub-sub messaging and fan-out notifications for user-facing and operator-facing alerts \cite{aws-sns}. It was selected to keep the notification path simple and loosely coupled, especially for events such as automation activity, important state changes, and alerts. SNS works well here because notifications are part of the overall platform, but they are not the same as workflow orchestration itself. The trade-off is that more complex automation logic still belongs in Step Functions or Lambda rather than in the notification layer.
