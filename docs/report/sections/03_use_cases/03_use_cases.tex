\section{Use Cases and Actors}

The system is composed of three core actors -- the \textbf{Consumer}, a user of the system, the \textbf{Device Controller}, responsible for orchestrating control of the devices, and the \textbf{Platform Operator}, responsible for maintaining services.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.6\linewidth]{sections/03_use_cases/conops.png}
  \caption{System Conops}
  \label{fig:conops}
\end{figure}

In a typical operational profile of this system, a consumer will purchase several devices and connect them to the internet via their personal wifi. The consumer will interact with the devices remotely either through an app on their phone or a web interface. The consumer is assumed to be non-technical, and will perform operations such as locking doors, viewing camera streams, and configuring rules such as locking schedules or thermostat schedules. The device controller, installed on each physical hardware system integrated within the system will be designed by a third party, and will be responsible for connecting to wifi or bluetooth and being available for connection with the cloud platform. The platform operator will be responsible for observing operational logs and remediating coverage issues, such as bugs in the system, service outtages, or delivering feature requests to enable future expansion of the system.


\subsection{Workflows}

The sequence diagrams corresponding to the following use cases are provided in Appendix~\ref{sec:appendix-a1}.

\begingroup
\setlength{\parindent}{0pt}

\textbf{UC-01: Remote Device Control.} A Consumer opens the frontend application, authenticates, and issues a command to a registered module such as locking a door or turning off a light. The platform validates the request, delivers the command to the target Device Controller, and reflects the confirmed state change in the dashboard in near-real time.

\textbf{UC-02: Remote Camera Monitoring.} A Consumer requests either a live stream or a recording from a specified time window via the frontend application. The platform retrieves and returns a playback-ready resource, driving the media ingestion, storage, and retrieval requirements of the architecture.

% This use case might be out of scope for the project
\textbf{UC-03: Automation Rule Programming.} A Consumer defines trigger-action rules in the frontend for example, locking all doors at 10:00 PM or activating lights when motion is detected and the platform persists and registers them for evaluation. At runtime, the platform evaluates active rules against incoming device events and scheduled triggers, dispatching commands to the appropriate Device Controllers when conditions are met.

\textbf{UC-04: Device Registration and Household Setup.} When a Consumer powers on a new module, it connects to the household WiFi and initiates a registration handshake with the platform, which assigns the device a unique identity and associates it with the Consumer's household. This use case also covers initial household onboarding, including account creation and WiFi credential provisioning.

\textbf{UC-05: Event Notification and Alerting.} A Consumer configures alert preferences in the frontend, specifying which event types and modules should trigger notifications and which delivery channels push, SMS, or email to use. When a qualifying event is published by a Device Controller, the platform evaluates it against the Consumer's preferences and dispatches an alert through the appropriate channel.

% This use case might be out of scope for the project
\textbf{UC-06: Access Delegation.} A Consumer invites a secondary household member via the frontend, assigning a scoped permission profile that restricts their access to designated modules and operations. The platform's authorization model supports per-module, per-user grants that the primary account holder can modify or revoke at any time.

\textbf{UC-07: Historical Event and Activity Review.} A Consumer queries the event log in the frontend by device, event type, or date range, and the platform returns matching records along with any associated media clips. This use case drives the data retention and indexing requirements of the storage layer, as Consumers expect to query history spanning days to months with acceptable response times.

\textbf{UC-08: Platform Operations and Fleet Monitoring.} A Platform Operator monitors service health, device connectivity, and business metrics through internal dashboards, and is alerted by the platform when threshold breaches such as elevated device connection failures require investigation. This use case also covers deployment of platform software updates and issuance of firmware update jobs to Device Controllers across the fleet.
