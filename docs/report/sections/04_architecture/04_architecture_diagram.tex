\subsection{AWS Service Descriptions and Responsibilities}

This architecture separates the platform into three major paths: a web and API ingress path for end users, a device connectivity path for smart-home modules, and a camera media path for security video. The intent of this separation is to keep the design high level and operationally clear while still mapping each major requirement to a managed AWS service. User-facing HTTPS traffic enters through CloudFront and API Gateway, device control and telemetry traffic terminate at AWS IoT Core, and camera video is handled through Amazon Kinesis Video Streams rather than being forced through the general IoT messaging path.

\subsubsection{Web and API Ingress}

\textbf{Amazon CloudFront} serves as the public edge entry point for the web application and distributes static frontend assets globally. By caching the user interface close to end users, CloudFront reduces latency for household members accessing the platform remotely and limits repeated origin requests for static content. The frontend itself is hosted in Amazon S3 and delivered through CloudFront rather than from a custom web server, which keeps the presentation layer simple and highly available.

\textbf{AWS WAF} protects the internet-facing application surfaces attached to CloudFront and Amazon API Gateway. WAF enforces managed and custom filtering rules that block common web attacks, abusive request patterns, and excessive request rates before those requests reach the application layer. At this level of design, WAF represents the primary web traffic protection boundary rather than a standalone application component.

\textbf{Amazon API Gateway} exposes the HTTPS API consumed by the web and mobile clients. It acts as the managed boundary between authenticated users and backend business logic, handling request routing, throttling, CORS, and token validation so those concerns do not need to be reimplemented in the application layer. API Gateway is used for operations such as onboarding, household management, device control requests, automation configuration, and retrieval of current device state or telemetry summaries.

\textbf{Amazon Cognito} provides the user identity and authentication layer for the consumer-facing platform. It manages user registration, sign-in, password policies, and multi-factor authentication, and issues tokens that are validated before protected API calls are allowed to proceed. Cognito is intentionally focused on end-user identity; device identity and authorization remain part of the IoT connectivity path rather than being merged into the same service boundary.

\subsubsection{Device Connectivity and Media Services}

\textbf{AWS IoT Core} is the primary connectivity service for smart-home devices such as lights, locks, thermostats, sensors, and camera control channels. Devices connect to IoT Core over secure device protocols, allowing the platform to receive telemetry, publish remote control commands, and maintain a managed message path without exposing internal services directly to device traffic. At this high level, IoT Core represents the device ingress, secure messaging, and cloud-side integration point for modular household devices.

\textbf{AWS IoT Device Management} supports large-scale onboarding and lifecycle operations for deployed devices. It provides a high-level administrative plane for device registration, grouping, inventory management, and over-the-air update workflows, which makes it a strong fit for the requirement that new modules be added with minimal end-user friction and that software or firmware updates be supported over time. Including Device Management in the architecture makes the onboarding and maintainability story explicit without forcing the diagram into lower-level provisioning details.

\textbf{Amazon Kinesis Video Streams} provides the dedicated media path for security cameras. Camera devices still use AWS IoT Core for control, health, and event signaling, but the actual video stream is published to Kinesis Video Streams rather than being handled as ordinary IoT telemetry. This separation allows the architecture to support near-real-time or live camera viewing in a way that is better aligned with video workloads. When an end user requests a camera stream, the application authorizes access and returns the necessary playback session information, after which the client consumes the stream directly from Kinesis Video Streams.

\subsubsection{Application and Automation Services}

\textbf{AWS Lambda} hosts the core backend application logic for the platform. Lambda functions process authenticated API requests, coordinate device control operations, read and update household and device metadata, evaluate or trigger automations, and prepare data returned to the user interface. Lambda also acts as the primary integration point between the web/API path and the device path by translating application requests into IoT operations and by transforming telemetry and media metadata into forms suitable for end-user consumption.

\textbf{AWS Step Functions} orchestrates multi-step workflows that should not be modeled as a single function invocation. This includes user-defined automations, conditional event responses, onboarding flows that require several coordinated steps, and escalation scenarios such as security alerts that involve multiple downstream actions. Step Functions makes the automation story easier to explain architecturally because it separates workflow state management from the stateless compute performed in Lambda.

\textbf{Amazon SNS} delivers notifications generated by backend workflows and device events. Consumer-facing examples include door unlock alerts, camera motion notifications, and reminders triggered by automation rules. SNS is also a convenient target for operational alarms originating from CloudWatch, allowing the same service family to support both end-user messaging and platform alerting while still keeping those topics logically separate.

\subsubsection{Data Storage}

\textbf{Amazon DynamoDB} stores the platform's primary operational data, including user and household records, device registrations, room assignments, automation rule definitions, notification preferences, and other application metadata. DynamoDB is a good fit for the high-volume, low-latency access patterns expected from a consumer platform that must scale across thousands of independently deployed home environments.

\textbf{Amazon Timestream} stores time-series telemetry produced by connected devices. Typical examples include temperature readings, motion events, battery levels, connectivity status, and other measurements that are naturally queried over a time window. Separating telemetry history into Timestream keeps the operational metadata model in DynamoDB focused on current state and configuration rather than long-running event history.

\textbf{Amazon S3} serves multiple storage roles in the architecture. It stores the static frontend assets that are distributed through CloudFront, and it also provides durable object storage for exported camera media, snapshots, archival artifacts, and other large or infrequently accessed objects. By using S3 for these storage concerns, the architecture avoids pushing binary media and static asset storage into services that are better suited for transactional or time-series workloads.

\subsubsection{Monitoring and Observability}

\textbf{Amazon CloudWatch} is the central observability service for the platform. It collects logs, metrics, and alarms across API Gateway, Lambda, AWS IoT Core, Step Functions, and Kinesis Video Streams, giving operations teams a unified view of platform behavior. CloudWatch alarms can publish to Amazon SNS for operational notifications when thresholds related to availability, error rates, or device connectivity are breached.

\textbf{Architectural Flow Summary.} End users access the application through CloudFront, protected by AWS WAF, and invoke backend operations through API Gateway with authentication handled by Amazon Cognito. Backend requests are processed by AWS Lambda, which reads and writes operational state in DynamoDB, coordinates workflows through Step Functions, and sends consumer notifications through SNS. Smart-home modules connect directly to AWS IoT Core for command, control, and telemetry, while AWS IoT Device Management handles lifecycle functions such as onboarding and over-the-air updates. Security cameras add a parallel media path through Amazon Kinesis Video Streams, with archival or exported objects stored in Amazon S3 and service health monitored through CloudWatch.
