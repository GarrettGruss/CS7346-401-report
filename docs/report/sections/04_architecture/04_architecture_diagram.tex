\section{System Architecture Diagram}
\label{sec:architecture}

Figure~\ref{fig:architecture} presents the proposed AWS architecture for \textit{LightsPlease}. The design separates three different traffic patterns as user facing web and API traffic, device command and telemetry traffic, and camera media traffic. This separation aligns with the kata's requirement for the platform to support modular smart home units, remote Internet access, and customer defined automations without forcing every workload through the same technical path. This architecure assumes the user already has a Wifi connection for web enabled devices, since the modules only need secure outbound connectivity to AWS rather than a dedicated on-premises managed local server. Protocol hubs such as Phillips Hue that establish a ZigBee network rather than Wifi can still be used, while not required, since the hub can directly connect to AWS IoT Core over the household network.

\subsection{Web and API Ingress}

Amazon Cognito is used to first authenticate users, who can then interact with the platform through a protected web and API boundary built around CloudFront, AWS WAF, and API Gateway. This provides an ingress layer that gives the system a clean Internet facing edge for HTTPS traffic, reduces direct exposure of backend services, and supports the kata's requirement for customers be able to monitor and control their homes remotely. Cognito provides identity tokens to a users web or mobile clinet, these tokens allow the API Gateway to be called, and then allow Lambda to enforce household specific authorization before executing device control, automation, or video playback requests. This provides an additional layer of security and protection to the user, since this platform prevents the user from needing to expose devices on their private network to the internet. 

\subsection{Device Control and Automation Path}

The primary control plane for non-video modules is centered on AWS IoT Core. This architecture assumes a lightweight MQTT over TLS model for lights, locks, thermostats, and similar units. That protocol choice is intentional as it is better suited to constrained, intermittently connected devices than pushing every interaction through a conventional request response API \cite{webbylab}. Devices connect through the customer's home network, publish telemetry and status updates to IoT Core, and receive cloud originated commands over the same secure channel. Lambda provides the higher level application logic around device registration, command validation, household ownership, and automation dispatch, while Step Functions coordinates longer running or multistep workflows such as scheduled routines, chained actions across several modules, or recovery logic when part of an automation fails.

\subsection{Camera and Media Path}

Cameras use the same general cloud control model for registration, health reporting, and user authorization, but their media path is intentionally separated from ordinary IoT messaging. This is due to IoT Core not being designed to handle basic control data, and telemetry. For this, Amazon Kinesis Video Streams are utilized to handle the video encoding and streaming of camera feeds. This is one of the most important design decisions in the architecture because remote camera observation is a core kata requirement, yet video has very different bandwidth, retention, and playback characteristics than simple device telemetry. Keeping video on a dedicated streaming path prevents that traffic from overwhelming the control plane and keeps the architecture easier to scale as camera adoption grows. Amazon S3 complements this path by storing larger objects such as snapshots, exported clips, or retained media artifacts when object storage is more appropriate than an active stream.

\subsection{Data Collection and State Management}
The data layer is divided by workload rather than collapsed into a single database. DynamoDB stores operational state that must be read and updated quickly, including household records, registered devices, automation definitions, command metadata, and current device status. Timestream stores time-series telemetry such as temperature readings, power measurements, motion events, or historical sensor data that may later be queried for dashboards, trends, or automation conditions. This separation was utilized to keep the platform more modular and support scalability. This allows potential future devices to utilize a common control architecture while still allowing flexibility in data management, as well as preventing a monolithic data storage solution.

\subsection{Monitoring, Notifications, and Architectural Fit}

CloudWatch provides the common observability layer across the public APIs, Lambda functions, automation workflows, and device integrations. SNS is utilized to handle user facing or operator facing notifications when an automation fires or when the platform detects an important event. The architecture also supports scalability, availability, and fault tolerance by incorporating a modular design that aims to utilize AWS Platform as a Service offerings. Most of the platform is built from PaaS offerings that distribute across AWS infrastructure with stateless logic. This reduces operational overhead for the platform, allows it to be securely accessed over the Internet, and responsive as the installed base grows over the first several years.
